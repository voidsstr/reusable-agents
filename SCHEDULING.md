# Agent Scheduling

The reusable-agents framework runs scheduled agents via **systemd `--user`
timers** on the host. There is one canonical mechanism — the same one
applies whether you write a Python `AgentBase` subclass, a bash entrypoint,
or anything else.

## Source of truth: `manifest.json`

Every agent has a `manifest.json`. Its `cron_expr` field declares when
the agent should run, and `entry_command` declares the shell command
that gets executed.

```json
{
  "id": "my-agent",
  "name": "My Agent",
  "category": "research",
  "task_type": "desktop-task",
  "cron_expr": "0 5 * * *",
  "timezone": "America/Detroit",
  "enabled": true,
  "entry_command": "bash /path/to/run.sh",
  "owner": "you@example.com"
}
```

Cron expressions are standard 5-field crontab syntax (`min hour dom mon dow`).
The framework converts the cron expression into a systemd `OnCalendar=` value,
honoring the manifest's `timezone`.

Manual-only agents (no cron) leave `cron_expr` blank — the framework still
registers them so they can be triggered ad-hoc via the dashboard's "Run now".

## Lifecycle

```
┌─────────────────────────────────────────────────────────────────────────┐
│ register-agents.sh                                                       │
│   POST manifest → /api/agents/register                                   │
│       └─ writes registry/agents.json in storage                          │
│       └─ returns                                                         │
└─────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ install/write-systemd-timers.py  (runs on the HOST)                      │
│   reads /api/agents/{id} for each registered agent                       │
│   for each with cron_expr + entry_command:                               │
│     ~/.config/systemd/user/agent-<id>.service  (ExecStart=...)           │
│     ~/.config/systemd/user/agent-<id>.timer    (OnCalendar=...)          │
│     systemctl --user daemon-reload                                       │
│     systemctl --user enable --now agent-<id>.timer                       │
└─────────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                  systemd fires the .timer
                              │
                              ▼
                  systemd-run start agent-<id>.service
                              │
                              ▼
                       entry_command runs
```

### Why a separate host-side script?

The framework API runs in Docker. Calling `systemctl --user daemon-reload`
inside the API container can't affect the host's systemd. Instead, the host
runs `install/write-systemd-timers.py` after registration — it imports
`framework.core.scheduler` directly so the unit content stays in lockstep
with what the API would write.

## Registering an agent

From the directory containing the manifest:

```bash
FRAMEWORK_API_URL=http://localhost:8093 \
FRAMEWORK_API_TOKEN=$FRAMEWORK_API_TOKEN \
bash /home/voidsstr/development/reusable-agents/install/register-agent.sh .
```

For a whole tree of manifests (e.g. `nsc-assistant/agents/`):

```bash
FRAMEWORK_API_URL=http://localhost:8093 \
bash /home/voidsstr/development/nsc-assistant/scripts/register-agents.sh
```

After registering, write the timers and enable them:

```bash
python3 /home/voidsstr/development/reusable-agents/install/write-systemd-timers.py
```

Idempotent — re-run any time after editing a manifest.

## Service unit content

Auto-generated by `framework/core/scheduler.py`. Looks like:

```
[Unit]
Description=Agent <id> (auto-wired by reusable-agents framework)
After=network.target

[Service]
Type=oneshot
WorkingDirectory=<repo_dir>
ExecStart=<entry_command>
Environment=AGENT_ID=<id>
Environment=AGENT_TRIGGERED_BY=cron
Environment=STORAGE_BACKEND=azure
Environment=AZURE_STORAGE_CONNECTION_STRING=<from-env-at-write-time>
Environment=AZURE_STORAGE_CONTAINER=agents
StandardOutput=append:/tmp/reusable-agents-logs/agent-<id>.log
StandardError=append:/tmp/reusable-agents-logs/agent-<id>.log

[Install]
WantedBy=default.target
```

The `Environment=AZURE_STORAGE_*` lines are populated from whatever was set
in the shell that ran `write-systemd-timers.py`, so the entry command (and
any subprocesses it spawns — implementer dispatches, etc.) inherit the
storage backend without each script needing to source `.env`.

## Timer unit content

```
[Unit]
Description=Schedule for agent <id>

[Timer]
OnCalendar=<derived from cron_expr + timezone>
Persistent=true
Unit=agent-<id>.service

[Install]
WantedBy=timers.target
```

`Persistent=true` means a timer that should have fired while the host was
off will fire on next boot.

## Triggering a run from the dashboard

Click "Run now" in the framework UI, or:

```bash
curl -X POST -H "Authorization: Bearer $FRAMEWORK_API_TOKEN" \
    http://localhost:8093/api/agents/<id>/trigger
```

The API writes a JSON file to `/tmp/agent-trigger-queue/`. The host-worker
(installed via `bash install/install-host-worker.sh`) polls that dir and
launches the agent's `entry_command` directly (bypassing systemd, which is
useful for ad-hoc runs that shouldn't reset the timer's `Persistent=true`
clock). Status is recorded in Azure Blob the same way as a cron-fired run.

## Verifying consistency

```bash
# How many agents declare cron?
curl -s -H "Authorization: Bearer $FRAMEWORK_API_TOKEN" \
    http://localhost:8093/api/agents | jq '[.[] | select(.cron_expr != "")] | length'

# How many timers exist on the host?
ls ~/.config/systemd/user/agent-*.timer | wc -l

# These two numbers should match. Investigate if they don't.
```

`install/write-systemd-timers.py` is idempotent — running it after any
manifest change re-syncs both directions (writes missing units, leaves
existing ones; orphan units must be cleaned up manually with
`systemctl --user disable --now agent-<id>.timer && rm ...`).

## Disabling / re-enabling an agent

Toggle `enabled: false` in the manifest, re-register, then:

```bash
systemctl --user disable --now agent-<id>.timer
```

Or via the dashboard — clicking the disable toggle calls
`POST /api/agents/<id>/disable` which updates the manifest in storage,
but you still need to run `disable --now` on the host to stop the timer.
(A future improvement is for the host-worker to reconcile timer state.)
