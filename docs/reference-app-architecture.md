# Reference App Architecture

> **Audience:** the `app-store-opportunity-agent` (when generating
> blueprints) and the `implementer` (when executing builds). This doc is
> the canonical "what every new app we ship looks like" reference. It
> mirrors `aisleprompt`'s production stack — that's our reference impl,
> verified at scale.

When the opportunity agent writes a blueprint, it includes a copy of
this doc's *Summary* section in the prompt context so the LLM grounds
its design in our actual stack. When the implementer picks up
`rec_type: app-build-from-blueprint`, it reads this doc end-to-end
before scaffolding.

---

## Summary (~400 words — meant for LLM prompt context)

Every new app ships as a **single full-stack monorepo** with five
artifacts:

1. **Web app** — React 18 + TypeScript + Vite (SPA with pre-rendering for
   SEO), Tailwind + HeadlessUI for styling, React Router for navigation,
   TanStack React Query for server state, Axios for API calls. Path:
   `frontend/`. Auth via Google OAuth (`@react-oauth/google`). Local dev
   port 3000; container port 4001 (nginx).
2. **Mobile app** — React Native via Expo SDK 54, TypeScript, React
   Navigation (bottom tabs + native stack), NativeWind (Tailwind on
   RN), AsyncStorage for local persistence. Path: `mobile/`. Auth via
   Google sign-in + Apple SignIn. Built via EAS, shipped to App Store
   + Play Store. Same backend API as web; localStorage-style guest
   mode for first-launch.
3. **Backend / API** — Node 18 + Express 4 + TypeScript, monolithic
   single-file server (`src/simple-server.ts` pattern) with all routes
   inline. Auth: JWT + passport-google-oauth20 with an
   `optionalAuthMiddleware` so guest endpoints work. Rate limit
   `express-rate-limit` (1000 req/min default). Body limit 20MB
   (image uploads). WebSocket via `ws` for live features. LLM calls
   route through Azure OpenAI primary + Ollama fallback (or use the
   framework's `ai_providers` chain).
4. **Database** — PostgreSQL 15 with **Prisma 5** as the ORM and schema
   manager. `prisma/schema.prisma` is the source of truth; migrations
   via `prisma migrate dev` for local, `prisma migrate deploy` for prod.
   Seeds in `src/seed.ts`. Connection via `DATABASE_URL` env. Local
   dev runs Postgres in docker-compose; production uses managed
   Postgres (Azure Database for PostgreSQL or equivalent).
5. **Agents (optional but encouraged)** — at least one
   reusable-agents framework instance per app, for SEO auditing,
   catalog quality, user-growth strategy, competitor research, etc.
   Each agent ships its own `manifest.json` + `site.yaml` and registers
   with the framework. Agents auto-queue recs to the implementer
   without email-reply gates by default.

**Delivery:** one `Dockerfile.azure` builds frontend → bundles backend
→ runs as a single container on **Azure Container Apps**. Local dev
uses `docker-compose.yml` with separate `db`, `backend`, and `frontend`
services. CI via GitHub Actions (lint → typecheck → test → image →
deploy).

**Shared code:** `packages/shared/` holds TypeScript types + small
utilities consumed by web, mobile, AND backend. No npm publish — the
monorepo's tsc-built `dist/` is referenced directly.

## Layer specs

### Web (frontend/)

```jsonc
// package.json — pinned versions
{
  "name": "<app>-frontend",
  "scripts": {
    "dev":     "vite",
    "build":   "tsc && vite build",
    "preview": "vite preview",
    "test":    "jest"
  },
  "dependencies": {
    "react":              "^18.2",
    "react-dom":          "^18.2",
    "react-router-dom":   "^6.14",
    "@tanstack/react-query": "^4.29",
    "axios":              "^1.6",
    "@react-oauth/google":"^0.12",
    "@headlessui/react":  "^1.7",
    "@heroicons/react":   "^2.0"
  },
  "devDependencies": {
    "vite":               "^5.2",
    "@vitejs/plugin-react": "^4.2",
    "typescript":         "^5.4",
    "tailwindcss":        "^3.4",
    "postcss":            "^8.4",
    "autoprefixer":       "^10.4"
  }
}
```

- **State management:** TanStack React Query for everything backend-fetched
  (auto-cache, dedupe, retry). React Context only for auth + theme.
- **Auth flow:** Google OAuth → backend `/api/auth/google` → JWT issued →
  stored in localStorage + sent on every Axios request.
- **Routing:** React Router with `<Routes>` at `App.tsx`; lazy-load
  pages with `React.lazy` for chunked bundles.
- **Pre-render:** for SEO, `prerender.js` runs Chromium against the
  built bundle and writes static HTML for crawled routes.

### Mobile (mobile/)

```jsonc
// package.json
{
  "name": "<app>-mobile",
  "scripts": {
    "start":  "expo start",
    "ios":    "expo run:ios",
    "android":"expo run:android",
    "build":  "eas build"
  },
  "dependencies": {
    "expo":               "~54.0",
    "react":              "19.1",
    "react-native":       "0.81.5",
    "@react-navigation/native":          "^6.0",
    "@react-navigation/bottom-tabs":     "^6.0",
    "@react-navigation/native-stack":    "^6.0",
    "@react-native-async-storage/async-storage": "^2.2",
    "axios":              "^1.15",
    "nativewind":         "^4.2",
    "expo-vector-icons":  "*",
    "expo-apple-authentication": "*"
  }
}
```

- **Bundle ids:** `com.<app>.app` for both iOS + Android.
- **Auth:** Google sign-in + Apple SignIn (Apple is required by App
  Store policy if you offer Google).
- **Guest mode:** the very first launch shows core value WITHOUT login;
  data persists to AsyncStorage and migrates to the user's account
  when they sign in.
- **API parity:** every screen calls the same backend endpoints as the
  web app — DO NOT fork mobile-only endpoints unless absolutely
  necessary.

### Backend (src/)

```jsonc
// package.json
{
  "name": "<app>-backend",
  "scripts": {
    "dev":        "ts-node-dev --respawn --transpile-only src/index.ts",
    "build":      "tsc",
    "start":      "node dist/index.js",
    "migrate":    "prisma migrate deploy",
    "seed":       "ts-node src/seed.ts"
  },
  "dependencies": {
    "express":            "^4.18",
    "@prisma/client":     "^5.0",
    "pg":                 "^8.11",
    "passport":           "^0.6",
    "passport-google-oauth20": "^2.0",
    "jsonwebtoken":       "^9.0",
    "express-rate-limit": "^6.7",
    "ws":                 "^8.13",
    "cors":               "^2.8",
    "dotenv":             "^16.0"
  },
  "devDependencies": {
    "typescript":         "^5.3",
    "ts-node-dev":        "^2.0",
    "prisma":             "^5.0"
  }
}
```

- **Single-file server pattern:** start with all routes in
  `src/simple-server.ts`. Don't pre-split into `routes/`,
  `controllers/`, `services/` until the file exceeds ~2000 LOC. The
  monolith catches issues earlier and keeps onboarding fast.
- **Auth middleware:** `optionalAuthMiddleware` — looks for the JWT,
  populates `req.user` if valid, leaves `req.user = null` otherwise.
  Guest endpoints just check `req.user` and adapt.
- **Body limit:** `express.json({ limit: '20mb' })` — accommodates
  base64 image uploads (pantry scans, profile photos).
- **Rate limit:** apply globally at 1000 req/min; lower per-route for
  expensive endpoints (LLM, image gen).
- **WebSocket:** mount at `/ws`, share the same JWT auth.
- **LLM calls:** never hardcode a provider — wrap every call so
  `framework.core.ai_providers.chat_with_fallback` can route between
  Copilot / Azure OpenAI / Ollama based on `config/ai-defaults.json`.

### Database (prisma/)

```prisma
// prisma/schema.prisma
generator client {
  provider = "prisma-client-js"
  output   = "../node_modules/.prisma/client"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

// One file. All models inline. Use comments to section it.
// User-facing IDs are uuid; FKs are inline relations with onDelete.
// Indexes EVERYTHING you'll filter on; default to btree.
```

- **Migrations:** `npx prisma migrate dev --name <slug>` for local;
  `npx prisma migrate deploy` for prod (in the docker entry script).
- **Seeds:** `src/seed.ts` runs on first deploy via the entry script;
  guard with `if (await prisma.user.count() > 0) return;`.
- **Connection string:** `postgresql://USER:PASS@HOST:5432/DB?sslmode=require`
  for production; `?sslmode=disable` for docker-compose.
- **Naming:** PascalCase models, camelCase fields, `_id` suffix on FKs.

### Shared (packages/shared/)

```ts
// packages/shared/src/types/index.ts — exported to all 3 layers
export interface User {
  id: string;
  email: string;
  display_name: string;
  created_at: string;   // ISO 8601
}

// packages/shared/src/api/client.ts — Axios instance config (URL, auth)
// packages/shared/src/storage/adapter.ts — uniform get/set across web/mobile
```

- Keep this MINIMAL. Types + tiny pure helpers only. Anything stateful
  (auth flow, API specifics) lives in the consumer.
- Build via `tsc -p packages/shared/tsconfig.json` to `dist/`; consumers
  reference `@<app>/shared` via workspaces.

### Agents (agents/)

When the new app needs automation (it usually does after week 2):

- Use the `reusable-agents/blueprints/` library — pick the closest
  blueprint and run `install/create-agent.sh`.
- Drop the agent dir at `agents/<app>-<purpose>-agent/`.
- Site config (DB DSN, brand voice, URL patterns) goes in
  `site.yaml` next to `manifest.json`.
- Register: `bash agents/register-with-framework.sh`.
- Common starter set: `seo-opportunity-agent`, `catalog-audit-agent`,
  `progressive-improvement-agent`, `competitor-research-agent`.

### Infra

```yaml
# docker-compose.yml — local dev
services:
  db:
    image: postgres:15
    ports: ["5436:5432"]
    environment:
      POSTGRES_USER: <app>
      POSTGRES_PASSWORD: dev
      POSTGRES_DB: <app>
    volumes: [db_data:/var/lib/postgresql/data]
  backend:
    build: { context: ., dockerfile: Dockerfile }
    ports: ["4004:4000"]
    depends_on: [db]
    environment:
      DATABASE_URL: postgresql://<app>:dev@db:5432/<app>?sslmode=disable
  frontend:
    build: { context: ., dockerfile: Dockerfile.frontend }
    ports: ["4001:80"]
volumes:
  db_data:
```

```dockerfile
# Dockerfile.azure — production
FROM node:20-alpine AS frontend-build
WORKDIR /app
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ ./
RUN npm run build && npx prerender   # pre-render for SEO

FROM node:18-alpine AS backend
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY --from=frontend-build /app/dist ./public
COPY src/ ./src/
COPY prisma/ ./prisma/
RUN npx prisma generate && npx tsc
ENV NODE_ENV=production PORT=4000
EXPOSE 4000
CMD ["sh", "-c", "npx prisma migrate deploy && node dist/index.js"]
```

- **Image tag format:** `<registry>/<app>:YYYYMMDD-HHMM` (UTC).
- **Production target:** Azure Container Apps with managed Postgres as
  an external service.
- **Env vars (required):** `DATABASE_URL`, `JWT_SECRET`,
  `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `AZURE_OPENAI_API_KEY`,
  `AZURE_OPENAI_ENDPOINT`, plus app-specific keys.
- **Deploy ritual:** local docker build → smoke test (`/api/health`) →
  Playwright suite → `az containerapp update`. Don't skip the smoke
  test even if the build looks clean.

### CI (.github/workflows/)

```yaml
# .github/workflows/deploy.yml
on: { push: { branches: [main] } }
jobs:
  test:
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 18 }
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck
      - run: npm test
      - run: npx playwright install --with-deps
      - run: npx playwright test
  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: azure/login@v1
      - run: az acr build -t $ACR/<app>:$(date -u +%Y%m%d-%H%M) .
      - run: az containerapp update -g <rg> -n <app> --image ...
```

## Hard rules every blueprint must follow

1. **No skipped layers.** Even if the user says "I just want the web
   app," scaffold mobile + agents stubs. The cost of building them
   later is 10x the cost of doing them now.
2. **Auth on day one.** Google + Apple. Guest mode supported (data in
   localStorage / AsyncStorage, migrates on sign-in).
3. **Prisma is the schema source of truth.** No raw SQL migrations,
   no Supabase migrations, no hand-edits to the DB.
4. **One `Dockerfile.azure` per app.** Frontend + backend in one
   image. Don't ship two images unless you have a hard reason
   (separate scaling profile, etc).
5. **Tests before deploy.** Playwright suite passes; smoke endpoint
   responds 200. CI gate.
6. **Tailwind for both web and mobile.** NativeWind on RN matches the
   web's tokens (palette, type scale, spacing). Same design system
   across both surfaces.
7. **TypeScript everywhere.** No raw `.js` files in new code.
8. **Single repo, monorepo workspaces.** Web, mobile, backend, shared
   live under one `package.json` with `workspaces: ["frontend",
   "mobile", "src", "packages/*"]`.
9. **Env vars via `.env.example`.** Document every required var; ship
   placeholder values. Never commit secrets.
10. **CLAUDE.md per app.** Document the site-specific rules: brand
    voice, north-star metric, the deploy ritual, anything that
    diverges from this reference.

## How the opportunity agent uses this doc

In `_blueprint_prompt`, the agent embeds this doc's *Summary* +
*File-tree* sections as additional system context. The LLM produces
opportunity-specific specs (product-spec, ux-research, ux-spec, api,
data-model, implementation-plan) that conform to the structure here —
the implementer can scaffold from a known shape.

## How the implementer uses this doc

When the implementer picks up `rec_type: app-build-from-blueprint`:

1. Read `manifest.json` from the blueprint's storage path.
2. Read THIS doc end-to-end.
3. Read every blueprint file the agent generated.
4. Scaffold the file tree above into a new repo (or branch).
5. Build milestone-by-milestone per `implementation-plan.md`.
6. Smoke-test + deploy on each milestone.

The blueprint says WHAT to build for this specific opportunity. This
doc says HOW it should be structured.
